# План оптимизации производительности

> **Текущий статус:** Lighthouse ~25/100, LCP 98с, TBT 11с, FCP 37с
> **Главная проблема:** ВСЕ зависимости загружаются в начальном бандле. Нет code splitting.

---

## Приоритет 1 — КРИТИЧНО (максимальное влияние)

### 1.1. Code splitting роутов через React.lazy + Suspense

**Проблема:** Все 4 страницы импортируются статически в `router/index.tsx`. React Flow (348KB), CodeMirror (132KB), Recharts (1.1MB), Pyodide JS — всё в начальном бандле, даже если пользователь заходит на другую страницу.

**Решение:** Заменить статические импорты на `React.lazy()` + `Suspense` для каждого роута.

**Файлы:**
- `front/src/app/router/index.tsx`

**Ожидаемый эффект:** Начальный бандл уменьшится на ~1.5-2MB. Пользователь, заходящий на `/`, не будет загружать CodeMirror и Recharts. Пользователь на `/tasks` не будет загружать React Flow.

**Риски:** Минимальные. `Suspense` fallback — простой лоадер. Маршруты не меняются.

---

### 1.2. Настройка manualChunks в Vite config

**Проблема:** Все зависимости пакуются в один огромный чанк. Нет разделения на vendor-чанки.

**Решение:** Добавить `build.rollupOptions.output.manualChunks` в `vite.config.ts` для разделения зависимостей на отдельные чанки:
- `antd` — отдельно (3.3MB)
- `recharts` — отдельно (1.1MB)
- `framer-motion` — отдельно (413KB)
- `react-flow` — отдельно (348KB)
- `codemirror` — отдельно (132KB)
- `icons` — отдельно (@ant-design/icons 1.5MB)

**Файлы:**
- `front/vite.config.ts`

**Ожидаемый эффект:** Браузер сможет кэшировать vendor-чанки отдельно. При изменении кода приложения не нужно перезагружать vendor-чанки. Параллельная загрузка чанков.

**Риски:** Минимальные. Чистая конфигурация сборки.

---

### 1.3. Отложенная загрузка Recharts на странице /tasks

**Проблема:** `TasksStatsSnippet` импортирует `PieChart` из `recharts` на странице `/tasks`. Это тянет всю библиотеку Recharts (1.1MB) в начальный бандл ради одного маленького donut-графика.

**Решение:** Вариант A — сделать `TasksStatsSnippet` ленивым компонентом с `React.lazy`. Вариант B — заменить маленький donut на CSS/SVG без Recharts. Вариант C — вынести Recharts-зависимые компоненты в отдельные lazy-компоненты.

**Файлы:**
- `front/src/widgets/tasks-stats-snippet/ui/tasks-stats-snippet.tsx`
- `front/src/pages/tasks/tasks-page.tsx`

**Ожидаемый эффект:** Страница `/tasks` не будет загружать Recharts для маленького графика. Экономия ~1.1MB.

**Риски:** Низкие. Нужно убедиться, что график корректно отображается после lazy-загрузки.

## Приоритет 2 — ВЫСОКИЙ (значительное влияние)

### 2.1. Условная загрузка Pyodide только при необходимости

**Проблема:** `preloadPyodide()` вызывается в `AppConfig` для КАЖДОГО пользователя при загрузке приложения. Это скачивает 5MB+ (pyodide.asm.wasm 2.6MB, python_stdlib.zip 2.3MB, pyodide.asm.js 221KB) с CDN, даже если пользователь никогда не откроет страницу задачи.

**Решение:** Убрать preload из `AppConfig`. Загружать Pyodide только когда пользователь открывает страницу задачи (`/task/:id`). Можно использовать `requestIdleCallback` внутри `TaskPage` при монтировании.

**Файлы:**
- `front/src/app/index.tsx` — убрать preloadPyodide
- `front/src/features/run-python/model/use-pyodide.ts` — добавить preload при первом использовании
- `front/src/pages/task/index.tsx` — вызвать preload при монтировании

**Ожидаемый эффект:** Устраняет 5MB+ ненужных загрузок для пользователей, которые не решают задачи. Значительное улучшение FCP/LCP.

**Риски:** Средние. Первый запуск кода на странице задачи будет медленнее (нужно загрузить Pyodide). Но это приемлемо — пользователь ожидает задержку при первом запуске.

---

### 2.2. Добавить preconnect для CDN

**Проблема:** `cdn.jsdelivr.net` не имеет preconnect. При загрузке Pyodide браузер тратит время на DNS lookup + TCP handshake + TLS.

**Решение:** Добавить в `index.html`:
```html
<link rel="preconnect" href="https://cdn.jsdelivr.net" />
<link rel="dns-prefetch" href="https://cdn.jsdelivr.net" />
```

**Файлы:**
- `front/index.html`

**Ожидаемый эффект:** Экономия ~320ms при загрузке Pyodide (по данным Lighthouse).

**Риски:** Нулевые.

---

### 2.3. Lazy-загрузка ContentRenderer (Recharts в shared UI)

**Проблема:** `ContentRenderer` импортирует `recharts` на уровне модуля. Этот компонент используется повсеместно и тянет Recharts в любой бандл, где он используется.

**Решение:** Сделать `ContentRenderer` ленивым компонентом или вынести chart-блоки (`ChartBlockView`) в отдельный lazy-компонент.

**Файлы:**
- `front/src/shared/ui/content/content-renderer.tsx`

**Ожидаемый эффект:** Recharts не будет загружаться на страницах, где нет chart-блоков.

**Риски:** Средние. Нужно убедиться, что все типы контента корректно рендерятся.

## Приоритет 3 — СРЕДНИЙ (умеренное влияние)

### 3.1. Оптимизация @ant-design/icons

**Проблема:** Пакет `@ant-design/icons` весит 1.5MB. Хотя используются именованные импорты, в development-режиме Vite загружает весь пакет.

**Решение:** Убедиться, что в production-билде tree-shaking работает корректно. Проверить итоговый размер иконок в бандле. Если всё ещё большой — рассмотреть замену на `lucide-react` (уже установлен на проекте) для части иконок.

**Файлы:**
- Все файлы с импортами из `@ant-design/icons`

**Ожидаемый эффект:** Потенциальная экономия 200-500KB если часть иконок заменить на lucide-react.

**Риски:** Низкие. Lucide-react уже установлен, иконки визуально похожи.

---

### 3.2. Оптимизация Framer Motion

**Проблема:** Framer Motion (413KB) используется в shared-компонентах на каждой странице (GlassButton, GlassTopbar, AppHeader). Нельзя отложить загрузку.

**Решение:** Вариант A — убедиться что framer-motion в отдельном vendor-чанке (решается через manualChunks). Вариант B — для простых анимаций (hover, fade) заменить на CSS transitions. Вариант C — использовать `framer-motion/client` если доступен.

**Файлы:**
- `front/src/shared/ui/glass-button/glass-button.tsx`
- `front/src/shared/ui/glass-topbar/`
- `front/src/widgets/app-header/`

**Ожидаемый эффект:** Не уменьшит бандл, но улучшит кэширование (vendor-чанк будет кэшироваться отдельно).

**Риски:** Средние для варианта B — нужно аккуратно заменить анимации без визуальных регрессий.

---

### 3.3. Перенести prism-react-renderer в dependencies

**Проблема:** `prism-react-renderer` указан в `devDependencies` но импортируется в production-коде (`content-renderer.tsx`).

**Решение:** Перенести в `dependencies` в `package.json`.

**Файлы:**
- `front/package.json`

**Ожидаемый эффект:** Корректная сборка в production.

**Риски:** Нулевые.

---

### 3.4. Оптимизация данных задач (model/data.ts)

**Проблема:** `model/data.ts` в `entities/task` весит 278KB (данные всех задач). Загружается на каждой странице.

**Решение:** Если данные используются только на определённых страницах — загружать их лениво. Или разбить на чанки по темам.

**Файлы:**
- `front/src/entities/task/model/data.ts`
- `front/src/entities/task/model/use-task-by-id.ts`
- `front/src/entities/task/model/use-tasks-by-topic.ts`

**Ожидаемый эффект:** Экономия ~200-278KB на страницах, где не нужен полный список задач.

**Риски:** Средние. Нужно аккуратно рефакторить логику загрузки данных.

**Файлы:**
- `front/vite.config.ts`

**Ожидаемый эффект:** Более быстрая dev-сервер startup, лучшая производительность в development.

**Решение:** Убедиться что Google Fonts загружается с `display=swap` (уже есть в URL). Добавить `font-display: swap` в CSS.

**Файлы:**
- `front/index.html`
- `front/src/index.css`

---

### 4.3. Убрать неиспользуемые импорты

**Решение:** Пройтись по всем файлам и удалить неиспользуемые импорты. Особенно проверить `account-page.tsx` после рефакторинга.

6. **2.3** (ContentRenderer lazy) — убирает Recharts из shared

8. **4.1-4.4** — polish
